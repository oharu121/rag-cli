## Fixing Pylance Import Warnings in VS Code

### The Problem

When opening the project in VS Code, Pylance shows warnings like:

```
Import "langchain_community.document_loaders" could not be resolvedPylancereportMissingImports
```

This happens because VS Code defaults to a global Python interpreter that doesn't have the project's packages installed.

### Solution: `pyrightconfig.json`

Create a `pyrightconfig.json` in the project root:

```json
{
  "venvPath": ".",
  "venv": ".venv"
}
```

This tells Pyright/Pylance to look for packages in the `.venv` folder, regardless of which interpreter VS Code has selected.

### Why This Works

- `pyrightconfig.json` directly configures **Pyright** (the type checker that Pylance uses)
- It overrides the interpreter selection for **import resolution**
- Works cross-platform (uses relative paths)
- Editor-agnostic (works in VS Code, Neovim, etc.)

### Alternative Considered: `.vscode/settings.json`

```json
{
  "python.defaultInterpreterPath": "${workspaceFolder}/.venv/Scripts/python.exe"
}
```

This approach tells VS Code to auto-select the venv interpreter. However, in testing, `pyrightconfig.json` worked more reliably after a VS Code reload.

### Important Note

The `.venv` folder must exist. Users need to run:

```bash
uv venv .venv
uv sync
```

Then reload VS Code (`Ctrl+Shift+P` → "Developer: Reload Window").

### Summary

| Approach | Scope | Reliability |
|----------|-------|-------------|
| `pyrightconfig.json` | Pyright/Pylance only | Works after reload |
| `.vscode/settings.json` | VS Code interpreter | Less reliable in testing |
| Manual interpreter selection | Per-user | Works but annoying |

**Recommendation:** Use `pyrightconfig.json` for projects to ensure anyone who clones won't see import warnings (after setting up `.venv`).

You are absolutely right to notice the similarity. In fact, **Web Search *is* a form of RAG**.

When an LLM uses a search tool (like Google Search), it is performing **Agentic RAG** (or "On-Demand RAG"). It generates a query, retrieves external data, and uses that data to generate an answer.

So, if "Tool Use" can already do this, why do we build complex, standalone RAG pipelines (Vector Databases, embeddings, etc.)?

The answer comes down to **Scale**, **Cost**, and **The "Needle in a Haystack" Problem**.

### 1. The "Read" vs. "Search" Problem
You asked: *"Why not define tool use to let LLM read documents?"*

The problem is **you cannot "read" what you haven't "found."**

* **Scenario A (Small Scale):** You have 5 PDFs. You can absolutely just give the LLM a tool called `read_file()`. The LLM loops through them, reads them, and answers. No RAG needed.
* **Scenario B (Large Scale):** You have **1 million corporate documents** (terabytes of data).
    * You cannot feed 1 million documents into the LLM's context window (it’s too expensive and exceeds the limit).
    * The LLM cannot "guess" which of the 1 million files contains the answer.

To solve Scenario B, you need a system to **find the top 5 relevant pages** out of that million *before* the LLM reads anything. That system (Indexing + Retrieval) **is** RAG.

Even if you wrap it in a "tool," that tool essentially becomes a RAG engine.

### 2. Standard RAG vs. Agentic RAG (Tool Use)
There are two ways to implement this "retrieval," and this explains why standard RAG is still popular.

#### A. Standard RAG (The "Pipeline" Approach)
* **How it works:** You pre-fetch relevant data *before* the LLM even sees the prompt.
    * *User asks question* $\rightarrow$ *Search Database* $\rightarrow$ *Paste results into Prompt* $\rightarrow$ *Send to LLM*.
* **Why bother?**
    * **Speed:** It is a single pass. 200ms for search + 2s for generation.
    * **Cost:** You only pay for one LLM call.
    * **Control:** You (the engineer) control exactly what chunks are retrieved using math (embeddings), not relying on the LLM's "mood" to call a tool correctly.

#### B. Agentic RAG (The "Tool Use" Approach)
* **How it works:** You give the LLM a tool and let it decide.
    * *User asks question* $\rightarrow$ *LLM thinks* $\rightarrow$ *LLM calls Search Tool* $\rightarrow$ *Tool returns data* $\rightarrow$ *LLM reads data* $\rightarrow$ *LLM answers*.
* **Why bother?**
    * **Flexibility:** The LLM can search multiple times if the first result is bad.
    * **Reasoning:** It can perform "Multi-hop" reasoning (e.g., search for X, read it, realize it needs Y, search for Y).
* **The Downside:** It is **slower** (multiple round-trips) and **more expensive** (more tokens).

### 3. Summary: Why we still need "RAG"
You are technically implementing RAG in both cases. The industry distinction is usually:

| Feature | **Standard RAG** | **Tool Use (Agentic RAG)** |
| :--- | :--- | :--- |
| **Mechanism** | Hard-coded pipeline (Search then Generate) | Dynamic loop (LLM decides when to search) |
| **Best For** | Static knowledge bases (e.g., Company Wiki, Help Docs) | Complex research, Multi-step problems |
| **Latency** | Fast (Low latency) | Slow (High latency) |
| **"Search" Engine** | **Required** (Vector DB / Keyword Search) | **Required** (Google / Internal API) |

**Conclusion:**
We "bother" implementing RAG because **LLMs are processors, not storage.** To let an LLM "read" your private data, you must first build a search engine for that data. That search engine *is* the RAG system. Whether you connect it via a hard-coded pipeline (Standard RAG) or a function call (Tool Use) is just an implementation detail.
